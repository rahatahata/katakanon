# 02 — アーキテクチャ

本書は [01_requirements.html](./01_requirements.html) を実装するにあたり、MVP のシステム構成と技術スタックの正本を記す。

---

## 1. ゴール

- **リアルタイム同期** — 1 ルーム = 1 ゲーム、全クライアントがサーバで検証された同一のゲーム状態を参照する（要件 [§9](./01_requirements.html#s9)）。
- **サーバ所有の状態機械** — ターン、スコア、解答キュー、指摘・投票、お題秘匿などの確定処理はバックエンドで行う。
- **ポートフォリオ性** — Next.js、Fastify、Socket.IO、PostgreSQL、Prisma、Docker Compose、CI を使い、フルスタック実装力と設計判断を示す。
- **MVP** — アカウント必須としない。ルームコード／URL 参加、プレイヤーネーム・手描きアイコン（ロビー中編集）、再接続用 `participant_token` で識別・復帰する。

## 2. 採用技術

| 領域             | 採用技術                     | 役割                                                      |
| ---------------- | ---------------------------- | --------------------------------------------------------- |
| フロントエンド   | Next.js / React / TypeScript | 画面、状態表示、レスポンシブ UI、Socket.IO クライアント。 |
| バックエンド     | Fastify / TypeScript         | REST API、入力検証、権限検証、ゲーム操作コマンドの入口。  |
| リアルタイム通信 | Socket.IO                    | ルーム単位の状態配信、参加者ごとの秘匿配信、再接続。      |
| DB               | PostgreSQL                   | ルーム、参加者、ゲーム結果、必要最小限の履歴の保存。      |
| ORM              | Prisma                       | スキーマ、migration、型付き DB アクセス。                 |
| ローカル環境     | Docker Compose               | PostgreSQL を含む再現可能な開発環境。                     |
| CI               | GitHub Actions 等            | lint、typecheck、test、Prisma schema / migration 検証。   |

## 3. リポジトリ構成

MVP は TypeScript モノレポとして構成する。

| パス                 | 役割                                                               |
| -------------------- | ------------------------------------------------------------------ |
| `apps/web`           | Next.js アプリ。ルーム作成、参加、ロビー、ゲーム画面、結果表示。   |
| `apps/server`        | Fastify / Socket.IO サーバ。REST API、リアルタイム通信、DB 接続。  |
| `packages/shared`    | API・Socket.IO の共有型、表示名バリデーション等。                  |
| `packages/game-core` | 純粋なゲーム状態機械。ターン進行、得点、解答キュー等をテストする。 |

## 4. 論理構成

| レイヤー                | 役割                                                                                             |
| ----------------------- | ------------------------------------------------------------------------------------------------ |
| **クライアント（Web）** | UI、Canvas によるアイコン描画、ゲーム画面、Socket.IO クライアント。                              |
| **リアルタイム層**      | Socket.IO によるルーム単位のブロードキャスト、参加者ごとの状態配信。                             |
| **アプリケーション層**  | ルーム作成、参加者管理、ターン／スコア／解答キュー／指摘・投票フローの状態機械、バリデーション。 |
| **永続化層**            | PostgreSQL / Prisma によるルーム、参加者、結果の保存。ゲーム中状態はサーバメモリを主とする。     |

---

## 5. 状態管理の考え方

- **単一の真実の源泉** — ゲーム状態（現在ターン、説明者、お題 ID、使用済みお題、スコア、解答キュー、指摘・投票フェーズ、オンライン状態等）は **サーバ側**で保持し、イベントで更新する。
- **イベント駆動** — クライアント操作（解答ボタン、正解／不正解、スキップ、指摘、是認／否認、投票、ゲーム開始等）は **命令**としてサーバに送り、サーバが検証後に **状態スナップショット**を配信する。差分配信は最適化として扱い、MVP では必須としない。
- **参加者ごとの秘匿配信** — お題テキストは説明者の接続にのみ含める。説明者以外への状態配信では、お題テキストや推測可能な ID を送らない。
- **クライアント非決定** — クライアントは状態を表示し、操作を送信する。得点、ターン終了、投票確定などのゲーム結果はローカルでは確定しない。

---

## 6. 永続化と認証

- **ゲーム進行中状態** — MVP ではサーバメモリ上の状態を主とする。再起動時の進行中ルーム復元は MVP の必須要件にしない。
- **DB 保存対象** — ルーム、参加者、ゲーム開始・終了時刻、最終スコア、ゲーム結果、必要最小限の履歴を PostgreSQL に保存する。
- **お題リスト** — MVP ではサーバ側の固定 JSON / TypeScript データとして保持する。DB 化やユーザー独自お題は将来機能とする。
- **認証** — MVP ではユーザーアカウント認証を入れない。参加者はルームコード、プレイヤーネーム、`participant_token` で識別する。
- **将来拡張** — ユーザー認証を導入する場合は `users` テーブルを追加し、参加者や作成ルームと関連付ける。

---

## 7. 非機能

- **レスポンシブ** — PC・スマホブラウザ（要件 [§2.4](./01_requirements.html#s24)）。
- **音声** — MVP ではアプリ外（Discord 等）。将来のアプリ内音声通話は [06_future_features.md](./06_future_features.md) 参照。
- **スケーラビリティ** — 友人規模を想定し、初期は単一プロセス／単一リージョンで可。複数台対応は Socket.IO Redis adapter 等を将来検討する。
- **CI** — lint、typecheck、unit test、Prisma schema / migration 検証を自動化する。
