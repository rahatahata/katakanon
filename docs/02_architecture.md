# 02 — アーキテクチャ

本書は [01_requirements.md](./01_requirements.md) を実装するにあたり、システム構成の方針を記す。詳細は実装フェーズで更新する。

---

## 1. ゴール

- **リアルタイム同期** — 1 ルーム = 1 ゲーム、全クライアントが同一のゲーム状態を参照する（要件 [§9](./01_requirements.md#9-非機能技術方針概要)）。
- **MVP** — アカウント必須としない。ルームコード／URL 参加、プレイヤーネーム・手描きアイコン（入室後）で識別。

---

## 2. 論理構成

| レイヤー | 役割 |
|----------|------|
| **クライアント（Web）** | UI、Canvas によるアイコン描画、ゲーム画面、WebSocket クライアント。 |
| **リアルタイム層** | ルーム単位のブロードキャスト、状態配信（例: WebSocket）。 |
| **アプリケーション層** | ルーム作成、参加者管理、ターン／スコア／解答キュー／指摘フローの状態機械、バリデーション。 |
| **永続化（任意・MVP）** | お題ワードリスト、（必要なら）ルームメタデータ。ゲーム中状態はセッション優先でも可。 |

---

## 3. 状態管理の考え方

- **単一の真実の源泉** — ゲーム状態（現在ターン、説明者、お題 ID、スコア、解答キュー、指摘フェーズ等）は **サーバ側**で保持し、イベントで更新する。
- **イベント駆動** — クライアント操作（解答ボタン、正解／不正解、指摘、是認／否認、投票、ゲーム開始等）は **命令**としてサーバに送り、サーバが検証後に **状態スナップショットまたは差分**を配信する想定。

---

## 4. 非機能（再掲・実装時に数値化）

- **レスポンシブ** — PC・スマホブラウザ（要件 [§2.4](./01_requirements.md#24-プラットフォーム)）。
- **音声** — MVP ではアプリ外（Discord 等）。将来 WebRTC は [§11](./01_requirements.md#11-将来機能本仕様のスコープ外の候補) 参照。
- **スケーラビリティ** — 友人規模を想定し、初期は単一プロセス／単一リージョンで可。
