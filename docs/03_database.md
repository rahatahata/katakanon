# 03 — データベース・永続化

本書は [01_requirements.md](./01_requirements.md) と [02_architecture.md](./02_architecture.md) に基づき、保存するデータの論理モデルと永続化方針を記す。

MVP の DB は **PostgreSQL**、ORM / migration は **Prisma** を採用する。

---

## 1. 方針

- **MVP** — 厳密なアカウントモデルは持たない。ルーム・参加者は **セッション ID**、**表示名**、再接続用 `participant_token` で扱う。
- **ゲーム進行中の状態** — サーバメモリ＋Socket.IO ブロードキャストを主とする。ただし、再接続・ホスト移譲・投票確定・お題重複防止に必要な状態は、サーバ側の単一状態として保持する。
- **永続化対象** — ルーム、参加者、ゲーム開始・終了時刻、最終スコア、ゲーム結果、必要最小限の履歴を PostgreSQL に保存する。
- **過剰永続化を避ける** — MVP ではターン、投票、解答の全イベントを最初から完全保存しない。将来 `game_events` を追加できるよう、状態機械と保存処理の境界は分ける。

---

## 2. 論理エンティティ（案）

### 2.1 お題ワード（マスタ）

| 項目              | 説明                                   |
| ----------------- | -------------------------------------- |
| `word_id`         | 一意 ID                                |
| `surface`         | カタカナ表記のお題文字列               |
| `locale` / `tags` | 任意メタデータ（MVP では必須ではない） |

MVP は **サーバ側の固定リスト**（要件 [§8.1](./01_requirements.md#81-お題ワードリスト)）をリポジトリ内 JSON / TypeScript データとして保持する。クライアントへ全件配布しない。

お題マスタの DB 化、ユーザー独自お題リスト、共有お題リストは将来機能として扱う。

---

### 2.2 ルーム（セッション）

| 項目                    | 説明                                                                              |
| ----------------------- | --------------------------------------------------------------------------------- |
| `room_id` / `room_code` | 参加用コード・URL 解決                                                            |
| `host_participant_id`   | ホスト（移譲で更新）                                                              |
| `settings`              | ラウンド数、お題ソース種別 等（要件 [§6](./01_requirements.md#6-ルーム作成設定)） |
| `status`                | `lobby` / `in_game` / `finished` 等                                               |
| `created_at`            | ルーム作成時刻                                                                    |

---

### 2.3 参加者（プレイヤー）

| 項目                     | 説明                                                                                              |
| ------------------------ | ------------------------------------------------------------------------------------------------- |
| `participant_id`         | ルーム内一意                                                                                      |
| `participant_token_hash` | 再接続用 `participant_token` の検証情報（生トークンは保存しない方針）                             |
| `display_name`           | プレイヤーネーム（検証ルール [§7.8.2](./01_requirements.md#782-プレイヤーネームの入力規約)）      |
| `avatar_payload`         | 手描きアイコン画像（URL 参照 or BLOB、上限は [§8.2](./01_requirements.md#82-参加者プロフィール)） |
| `score`                  | 現在得点                                                                                          |
| `joined_at`              | 入室順。ホスト自動移譲の優先順に利用                                                              |
| `connection`             | オンライン／オフライン状態                                                                        |

---

### 2.4 ゲーム状態（ランタイム）

ターン・解答キュー・指摘フェーズ等は **サーバメモリ上の構造体**で持ち、必要に応じて Redis 等にシリアライズする設計も可。

| 概念             | 説明                                                                                                     |
| ---------------- | -------------------------------------------------------------------------------------------------------- |
| 順番・現在ターン | 説明者 ID、お題 ID、ラウンド内インデックス、使用済みお題 ID                                              |
| 解答キュー       | 参加者 ID の列。同一参加者は同時に 1 件まで（要件 [§7.5](./01_requirements.md#75-解答キューと正誤判定)） |
| 指摘・投票       | 排他ロック、指摘者 ID、是認／否認、オンライン投票権者、投票済み状態、投票保留状態                        |

---

### 2.5 ゲーム結果

| 項目          | 説明                                         |
| ------------- | -------------------------------------------- |
| `result_id`   | 一意 ID                                      |
| `room_id`     | 対象ルーム                                   |
| `scores_json` | 最終スコア・順位を保存する JSON              |
| `created_at`  | 結果保存時刻                                 |

MVP では、遊んだ結果を確認できる最低限の保存を優先する。詳細なターン履歴や投票履歴は将来の `game_events` で扱う。

---

## 3. Prisma モデル方針

初期実装では、少なくとも次の Prisma model を用意する。

| Prisma model  | 対応する概念                         |
| ------------- | ------------------------------------ |
| `Room`        | ルーム（セッション）                 |
| `Participant` | 参加者（プレイヤー）                 |
| `GameResult`  | ゲーム結果                           |

`RoomStatus` は `LOBBY` / `IN_GAME` / `FINISHED` を基本とする。
