# 04 — API・リアルタイム契約

本書は [01_requirements.md](./01_requirements.md) を実装するための **HTTP API** および **リアルタイム（例: WebSocket）メッセージ**のたたき台である。パス・フィールド名は実装時に確定する。

---

## 1. 原則

- **認証** — MVP はアカウントなし。ルーム作成・参加時に **表示名** を送り、サーバが `participant_id` と再接続用 `participant_token` を発行する想定。
- **権限** — ホストのみ **ゲーム開始**、ホスト移譲は要件 [§7.8](./01_requirements.md#78-ルーム参加ホスト) に従う。
- **秘匿** — お題テキストは説明者にのみ配信する。説明者以外には、お題テキストや推測可能な `word_id` を送らない。

---

## 2. HTTP API（案）

| メソッド | パス（例）             | 説明                                                                                                                   |
| -------- | ---------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `POST`   | `/rooms`               | ルーム作成。ボディ: `display_name`、ラウンド数等。レス: `room_code`, `join_url`, `participant_id`, `participant_token` |
| `GET`    | `/rooms/{code}`        | ルームメタの参照（存在確認、設定表示用）                                                                               |
| `POST`   | `/rooms/{code}/join`   | 参加。ボディ: `display_name`（必須）。レス: `participant_id`, `participant_token`                                      |
| `POST`   | `/rooms/{code}/resume` | 再接続。ボディ: `participant_token`。レス: 復帰した参加者情報と最新状態                                                |
| `POST`   | `/rooms/{code}/avatar` | アイコン画像のアップロード（ロビー中のみ）。要件 [§7.8.3](./01_requirements.md#783-手描きアイコンの扱い)               |
| `GET`    | `/words`               | （管理用）お題マスタ取得。MVP では通常クライアントに公開しない                                                         |

静的ファイル配信・CORS・レート制限はデプロイ環境に依存。

---

## 3. WebSocket（または同等）イベント（案）

### 3.1 クライアント → サーバ（コマンド）

| イベント名（例） | ペイロード概要                                         |
| ---------------- | ------------------------------------------------------ |
| `game:start`     | ホストのみ。ゲーム開始                                 |
| `host:transfer`  | ホストのみ。任意の参加者へホスト権限を移譲             |
| `answer:enqueue` | 解答ボタン                                             |
| `answer:judge`   | 説明者: `correct` / `incorrect`、対象 `participant_id` |
| `answer:skip`    | 説明者のみ。加減点なしでターンをスキップ               |
| `accuse:start`   | 指摘ボタン（先着ロック）                               |
| `accuse:respond` | 説明者: `admit` / `deny`                               |
| `vote:cast`      | 否認時の可決／否決                                     |
| `profile:update` | ロビー中のみ。アイコン更新                             |

### 3.2 サーバ → クライアント（状態配信）

| イベント名（例） | 内容                                                       |
| ---------------- | ---------------------------------------------------------- |
| `state:full`     | ルーム全体スナップショット（参加リスト、スコア、フェーズ） |
| `state:delta`    | 差分更新（任意最適化）                                     |

スナップショットに含める情報の例:

- 参加者一覧（名前・アイコン URL・スコア）
- 現在フェーズ（ロビー / ターン中 / 指摘中 / 投票中）
- 説明者 ID、**お題テキスト**（説明者の接続にのみ含める。説明者以外には `word_id` も含めて送らない）
- 解答キュー順
- 指摘・投票の途中状態（指摘者、オンライン投票権者、投票済み状況、保留状態）

## 4. エラー・バリデーション

- プレイヤーネーム — [§7.8.2](./01_requirements.md#782-プレイヤーネームの入力規約) を **サーバで再検証**。
- ゲーム開始 — ホストのみ、オンライン参加者 3 人以上、かつ参加者数 × ラウンド数分のお題を重複なしで確保できる場合のみ許可する。
- 解答キュー — 同一参加者が同時に 2 件以上並ぶ操作、指摘・投票中の解答操作を拒否する。
- 正誤判定 — 説明者のみ許可する。`correct` は加点後にターン終了、`incorrect` はキュー先頭を削除する。
- スキップ — 説明者のみ許可する。加減点なしでターン終了とする。
- 指摘の排他、是認／否認、投票確定条件 — サーバが **単一の状態機械**で検証し、不正な状態遷移を拒否する。
- プロフィール更新 — プレイヤーネーム変更は拒否する。アイコン更新はロビー中のみ許可する。
