# MVP と将来機能の分離設計

## 目的

既存の仕様書は MVP と将来機能が同じ本文中に併記されており、実装時に「今作る範囲」と「後で検討する範囲」を読み分ける必要がある。

本変更では、MVP の実装判断に使う文書を読みやすくし、将来機能は別文書へ集約する。

## 採用方針

`docs/01_requirements.md` は MVP 仕様を主本文とする。将来機能の詳細は `docs/06_future_features.md` に移し、MVP 本文には必要な短い参照だけを残す。

この方針により、MVP の実装者は `01_requirements.md` を上から読めば現在のスコープを把握できる。一方で、将来機能の背景や検討済みのルールは失わず、別文書で継続管理できる。

## 対象文書

- `docs/01_requirements.md`
- `docs/03_database.md`
- `docs/04_api.md`
- `docs/05_sitemap.md`
- `docs/06_future_features.md`（新規）

`docs/02_architecture.md` は MVP の構成方針が中心のため、大きな変更は不要とする。ただし将来 WebRTC 等への参照が重い場合は、`06_future_features.md` への短い参照に寄せる。

## 分離ルール

- MVP の挙動、制約、バリデーション、画面、API、状態管理は既存文書に残す。
- 将来機能の詳細仕様、検討案、未採用ルール、拡張用フィールドの説明は `06_future_features.md` に集約する。
- MVP 文書から将来機能へ触れる場合は、原則として「MVP では採用しない。詳細は `06_future_features.md`」のような短い参照に留める。
- 改訂履歴は、各文書で今回の分離を追記する。

## 将来機能として切り出す項目

- 2 人プレイモード
- ライフ制
- 説明時間・時間切れ
- 投票時間
- ユーザー独自お題リスト
- 長時間無操作時のアラート・自動不正解
- 文字起こし・カタカナ検出
- アイコン描画の省力化
- アプリ内音声通話
- 将来拡張用のデータ項目や API

## 非対象

- ゲームルール自体の再設計
- 技術スタックの決定
- 実装コードの追加
- MVP から機能を削る判断

## 確認方法

- `01_requirements.md` 単体で MVP のルールと進行が追えること。
- 将来機能の詳細が `06_future_features.md` にまとまっていること。
- MVP 文書内に、将来機能の長い仕様説明が残っていないこと。
- 文書間リンクが存在するファイル名と見出しを指していること。
