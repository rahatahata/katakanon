# ポートフォリオ向け MVP 実装アーキテクチャ設計

## 目的

本アプリは、カタカナ禁止ゲームの MVP を実装するだけでなく、Web 系転職のポートフォリオとしてフルスタックの実装力を示すことも目的とする。

そのため、単に最短で遊べる構成ではなく、フロントエンド、バックエンド、リアルタイム通信、DB、テスト、CI、ローカル開発環境、設計判断を読み取れる構成を採用する。

## 採用方針

MVP は `Next.js + Fastify + Socket.IO + PostgreSQL + Prisma` を採用する。

| 領域 | 採用技術 | 目的 |
| ---- | -------- | ---- |
| Frontend | Next.js / React / TypeScript | 画面設計、状態表示、レスポンシブ UI を実装する |
| Backend | Fastify / TypeScript | REST API、バリデーション、ゲーム操作コマンドの入口を実装する |
| Realtime | Socket.IO | ルーム単位の状態配信、参加者ごとの秘匿配信、再接続を扱う |
| Database | PostgreSQL | ルーム、参加者、ゲーム結果、最低限の履歴を保存する |
| ORM | Prisma | スキーマ、migration、型付き DB アクセスを明確にする |
| Local infra | Docker Compose | PostgreSQL を含むローカル開発環境を再現可能にする |
| CI | GitHub Actions 等 | lint、typecheck、test、migration 検証を自動化する |

## 選定理由

本プロジェクトでは、ゲーム進行の正しさとリアルタイム同期が中核になる。クライアント中心や BaaS 中心に寄せると、お題の秘匿、指摘の排他ロック、投票確定、解答キュー、再接続時の状態復元を一貫して説明しづらい。

そこで、ゲーム判定と状態遷移は自作バックエンドに集約する。一方で、永続化は PostgreSQL と Prisma を使い、実務で評価されやすい DB 設計と migration の流れを示す。

Next.js は UI と画面遷移の実装に使い、Socket.IO サーバとは分離する。Next.js custom server に寄せないことで、フロントエンドとバックエンドの責務を明確にし、デプロイや運用の説明もしやすくする。

## アーキテクチャ

### フロントエンド

Next.js アプリは、ルーム作成、ルーム参加、ロビー、ゲーム進行、結果表示を担当する。

クライアントはゲーム状態を直接確定せず、ユーザー操作を API または Socket.IO のコマンドとしてバックエンドへ送る。画面はサーバから配信されたスナップショットを表示する。

主な責務:

- プレイヤーネーム入力とルーム参加
- ロビーの参加者表示、手描きアイコン編集
- ゲーム中の現在フェーズ、説明者、解答キュー、指摘・投票状態の表示
- 説明者だけにお題を表示する UI
- 結果画面と順位表示

### バックエンド

Fastify アプリは REST API と Socket.IO サーバを提供する。ゲーム進行はサーバ側の状態機械で扱い、クライアントからの命令を検証してから状態を更新する。

主な責務:

- ルーム作成、参加、再接続
- `participant_token` の発行と検証
- ホスト権限の検証
- ゲーム開始条件の検証
- ターン、ラウンド、説明者順の管理
- 解答キュー、正誤判定、スキップ
- 指摘、是認・否認、投票、投票確定
- お題テキストの参加者別秘匿配信
- 終了結果の保存

### リアルタイム通信

Socket.IO の room をルーム単位の配信チャネルとして使う。

サーバは原則として状態スナップショットを配信する。最初は差分配信を必須にせず、実装の単純さと状態の追いやすさを優先する。

お題テキストは説明者の接続にだけ含める。説明者以外には、お題テキストや推測可能な `word_id` を送らない。

### DB と永続化

MVP のゲーム進行中状態はサーバメモリを主とする。DB には、MVP の価値とポートフォリオ上の説明力に必要な情報を保存する。

保存するもの:

- ルーム情報
- 参加者情報
- ゲーム開始・終了時刻
- 最終スコア
- ゲーム結果
- 必要最小限の履歴

最初から全ターン、全投票、全解答を厳密にイベント保存することはしない。ただし、将来 `game_events` を追加できるよう、状態機械と DB 保存処理の境界は分ける。

### 認証

MVP ではユーザーアカウント認証を入れない。

参加者はルームコード、プレイヤーネーム、サーバが発行する `participant_token` で識別する。`participant_token` は再接続用であり、生トークンは保存しない方針とする。

将来、ユーザーアカウントを導入する場合は、`users` テーブルを追加し、参加者や作成ルームと関連付けられるようにする。

## データフロー

1. ユーザーがルームを作成する。
2. バックエンドがルーム、ホスト参加者、`participant_token` を作成する。
3. フロントエンドが Socket.IO に接続し、ルームに join する。
4. サーバが現在の状態スナップショットを配信する。
5. ユーザー操作はコマンドとしてサーバへ送信される。
6. サーバが権限、フェーズ、入力値を検証する。
7. 検証成功時のみ状態機械を更新する。
8. サーバが参加者ごとに必要な情報を含めたスナップショットを配信する。
9. ゲーム終了時に結果を PostgreSQL へ保存する。

## エラー処理

サーバは不正な状態遷移を拒否し、クライアントには表示可能なエラーコードとメッセージを返す。

主なエラー:

- ルームが存在しない
- `participant_token` が無効
- ホスト以外がゲーム開始やホスト移譲を試みた
- ゲーム開始条件を満たしていない
- 現在フェーズで許可されない操作をした
- 解答キューに重複登録しようとした
- 説明者以外が正誤判定やスキップを試みた
- 投票権のない参加者が投票しようとした

## テスト方針

ポートフォリオとして、UI よりもまずゲーム状態機械のテストを厚くする。

優先して書くテスト:

- ルーム作成と参加
- ゲーム開始条件
- ターン順とラウンド終了
- 解答キューの重複防止
- 正解時の両者加点
- 不正解時のキュー処理
- スキップ時の無加点ターン終了
- 指摘の排他ロック
- 是認時と否認時の分岐
- 投票確定と同票否決
- お題が説明者以外に配信されないこと

E2E は MVP 後半で、ルーム作成からゲーム終了までの代表フローに絞る。

## CI と開発体験

CI では最低限次を実行する。

- lint
- typecheck
- unit test
- Prisma schema / migration の検証

ローカル開発は Docker Compose で PostgreSQL を起動し、アプリは通常の Node.js 開発サーバとして起動する。README には、初回セットアップ、migration、テスト、起動方法、主要な設計判断を記載する。

## 将来拡張

MVP の完成後、次の拡張を検討できる。

- Redis adapter による Socket.IO の複数台対応
- Redis へのゲーム中状態の退避
- `game_events` によるイベント履歴保存
- 画像保存先の外部 Storage 化
- ユーザー認証と作成ルーム履歴
- 管理用のお題リスト編集
- 監視、ログ、メトリクス

これらは MVP では実装しない。最初の目標は、遊べる完成品と、設計意図が読めるコードベースを作ることとする。

## 非対象

- MVP 時点での OAuth 等のユーザー認証
- Redis を使った複数台スケール
- 全イベントの完全な永続化
- アプリ内音声通話
- ユーザー独自お題リスト
- 文字起こしや自動カタカナ検出

## 確認方法

- 仕様書の MVP 範囲と矛盾しないこと。
- ゲーム判定がクライアントではなくサーバ状態機械に集約されていること。
- お題秘匿、解答キュー、指摘、投票、再接続を説明できる構成であること。
- PostgreSQL と Prisma を使う理由がポートフォリオ上の目的と対応していること。
- MVP で実装しない将来機能が、拡張点として分離されていること。
