# カタカナ禁止ゲーム — プロダクト仕様書（要件定義）

本ファイルは `docs/01_requirements.md` として管理する（旧 `SPEC.md`）。

---

## 1. 文書の目的



本書は、通話ツール（Discord 等）と併用して遊ぶ **カタカナ禁止説明ゲーム** を Web アプリとして実装するにあたり、プロダクトの範囲・挙動・用語を共有するための仕様である。



---



## 2. 概要



### 2.1 一言説明



参加者の一人が **カタカナのお題** を、**カタカナを使わず**通話で説明し、他の参加者が正解を当てる。違反疑惑は **指摘と投票** で処理し、**得点** で順位をつける **セッション型マルチプレイ** ゲーム。



### 2.2 アプリの担当範囲



| 担当 | 内容 |

|------|------|

| 本アプリ | ルーム管理、参加者同期、お題の秘匿表示、**参加者プロフィール**（**プレイヤーネーム**は入室時、**手描きアイコン**は入室後編集、[7.8.1](#781-入室時のプレイヤー設定)）、順番・ターン・ラウンド・得点、解答・正誤・指摘・投票の **手続きと記録** |

| 外部（通話ツール） | 音声による説明・発話・話し合い |

| 参加者 | 「カタカナを言ったか」等の **ルールの解釈**（システムは内容を定義しない。詳細は [4. ルールと判定](#4-ルールと判定)） |



### 2.3 目的・対象



- **主な利用者**: 開発者の友人同士での遊び。

- **開発者向け目的**: Web 技術の習得・発展、ポートフォリオとしての提示。



### 2.4 プラットフォーム



- **PC・スマートフォンの Web ブラウザ**から利用可能とする（レスポンシブ対応を前提とする）。

## 3. 用語



| 用語 | 意味 |

|------|------|

| ルーム | 1 つのゲームセッションの単位。**1 ルーム = 1 ゲーム**。 |

| 参加者 | ルームに参加しているプレイヤー。 |

| プレイヤーネーム | 入室時に参加者自身が入力する **表示名**。**必須**。検証ルールは [7.8.2](#782-プレイヤーネームの入力規約) を参照。 |

| アイコン（手描き） | **入室後**に **描画UI** で編集・確定する **アイコン**。未設定・空白でも **入室可能**（プレースホルダー表示）。詳細は [7.8.3](#783-手描きアイコンの扱い) を参照。 |

| ホスト | **ルーム作成者**。ゲーム開始・ホスト権限を持つ。切断時は権限が移譲され、**任意の参加者へ手動移譲**もできる（[7.8](#78-ルーム参加・ホスト））。 |

| 説明者 | 当該ターンにお題を知り、カタカナを使わず説明する役。 |

| 解答者 | 説明者以外。正解を推測し解答する。 |

| 指摘者 | 説明者のカタカナ使用を疑い、指摘操作を行った参加者。 |

| ターン | 1人の説明者が **1お題** について説明し、解答・指摘・正誤判定などが行われる単位。 |

| ラウンド | 参加者 **全員** が **それぞれ1回ずつ** 説明者を経験するまでのまとまり。**1 ラウンド = N ターン**（N = 参加者数）。ゲーム全体では、ルーム作成時に指定した **ラウンド数** だけこれを繰り返す。 |

| お題 | **当該ターンの**説明者の画面にのみ表示される **カタカナ表記のワード**（MVP は固定リスト）。 |

| 解答キュー | 解答ボタンが押された順に並ぶ **解答者の待ち行列**。先頭が「解答権」を持つ（[7.5](#75-解答キューと正誤判定)）。 |

## 4. ルールと判定



### 4.1 カタカナ禁止の性質



- ゲーム性は **外来語等を言い換える言葉遊び** にある。

- **違反の内容・境界**（何を「カタカナとみなすか」等）は **参加者同士の話し合い** に委ねる。**本システムはその内容を定義しない。**



### 4.2 システム上の違反処理

- 違反の **手続き的な認定** は、**指摘ボタン → 説明者の是認・否認 → （否認時のみ）投票** の結果として記録する（[7.4](#74-指摘と投票)）。



---



## 5. 参加者人数とモード



### 5.1 想定（本仕様の主対象）



- **オンライン同期の人数可変ルーム**（各人 **1 端末** が基本）。

- **投票機能**を用いる指摘フローのため、**最低 3 人**を推奨する（説明者・指摘者を除いた解答者が少なくとも 1 人投票に参加できる構成）。



### 5.2 将来検討



- **2 人プレイ**（投票なし、指摘の扱いを別ルール化する等）は **将来機能** とする。

## 6. ルーム作成・設定



ルーム作成時（または開始前）に、少なくとも次を指定できるものとする。



| 設定項目 | 説明 |

|----------|------|

| ラウンド数 | **全員が1周する**単位（[3. 用語](#3-用語) の **ラウンド**）を、ゲーム中に **何回** 繰り返すか。 |

| お題ソース | MVP は **固定リスト**。将来、ユーザー独自リストを追加可能とする。 |



**ライフ制**は **将来機能**（[11.1](#111-ライフ制将来)）。**MVP では採用せず**、順位は **得点のみ**（[7.9](#79-スコア順位)）で決定する。



**説明時間**（各ターンの **説明パート**に対する制限および **時間切れ**による強制進行）は **将来機能**（[11.2](#112-説明時間時間切れ将来)）。**MVP ではアプリによるタイマー・時間切れは設けない**（[7.6](#76-説明時間mvp)）。

## 7. ゲーム進行



### 7.1 全体フロー



1. **ルーム作成** — ホストが設定を入力し、参加用の **ルームコード** および **URL** の双方で参加できるようにする（[7.8](#78-ルーム参加・ホスト)）。

2. **参加者が合流** — 各参加者は **入室時** に [7.8.1](#781-入室時のプレイヤー設定) に従い **プレイヤーネーム** を入力して入室する。**手描きアイコン** は **入室後**に編集・確定する。ホストが **ゲーム開始** を操作する。

3. 各 **ラウンド** で、参加者全員が説明者となるまで **ターンを順に進める**。ルーム設定の **ラウンド数** だけこれを繰り返す。

4. 全ラウンド終了後、**得点に基づく順位**を表示する。



### 7.2 ラウンド・ターンと役割の決定



- **ラウンド** = 参加者 **全員** が **それぞれ1回ずつ** 説明者を経験するまでの期間。**1 ラウンド** は **N ターン**（N = 参加者数）で構成される。

- **ターン** = 1人の説明者が **1お題** を担当する区間。1ターン内では **説明者は1人**、**それ以外はすべて解答者**とする。

- **各ラウンド開始時**に、参加者の順番を **ランダム** 等で決定する（全員が1回ずつ説明者になる順序）。

- **各ターン開始時**に、順番に従い **次の** 参加者を **説明者** とし、新しい **お題** を説明者の画面にのみ表示する。



### 7.3 お題の表示



- **お題は当該ターンの説明者の画面にのみ**表示する。他参加者の画面では秘匿する。

- お題は **カタカナ表記のワード** とする（リストは MVP では固定）。



### 7.4 指摘と投票



- 解答者は、説明者がカタカナ語を使用したと判断した場合 **指摘ボタン** を押せる。

- **指摘ボタン**は **解答ボタン**（[7.5](#75-解答キューと正誤判定)）の **キュー方式**とは異なり、**排他ロック**とする。

  - **同一ターン内**で、**進行中の指摘フローがない**ときに限り、解答者は指摘ボタンを押せる。

  - **先着 1 人**のみが **指摘者**となり、**当該指摘フローが完了**するまで（**是認／否認**の表明、または **否認に伴う投票**の確定まで）、他の解答者は **指摘ボタンを利用できない**（押下を受け付けない、または無効とする）。

  - フロー完了後はロックが解除され、**同一ターン内**で **再度** 指摘が可能になる場合は、再び **先着 1 件のみ**（以降も同様）。

- 指摘が行われたら、**説明者**の画面に **是認**・**否認**の操作を表示する。

- **是認** — 説明者が、カタカナ使用を **認める**。**投票は行わない**。指摘が **成立**したものとして、**可決**と同様に **説明者から 1 点減点**（-1 pt）とする。

- **否認** — 説明者が **争う**。**説明者と指摘者を除く解答者**による **投票** に移行する。

- 投票で **可決**された場合: **説明者から 1 点減点**（-1 pt）。

- 投票で **否決**された場合: **指摘者から 1 点減点**（-1 pt）。



#### 7.4.1 投票が同票のとき



**可決票と否決票が同数**の場合は **否決扱い**とする（**指摘不成立**）。得点は上記の **否決**と同じく、**指摘者から 1 点減点**（-1 pt）。



### 7.5 解答キューと正誤判定



1. 解答者は **解答ボタン** を押すたびに **解答キュー** の末尾に追加される（**正誤確定待ちのあいだも押せる**）。

2. **解答権**は **キューの先頭**の解答者に付与される。先頭以外は、キューに並んでいる状態のみ（発言のタイミングは通話上のルールに委ねるが、アプリ上は「順番」を表示する）。

3. **説明者の画面にのみ**、現在の解答権保持者に対する **正解**・**不正解**の操作が表示される。

4. 説明者が **正解**を押した場合: **得点配分は [7.7](#77-得点配分と正解時の加点) に従う**（**解答者**と**説明者**の **両者加点**）。

5. 説明者が **不正解**を押した場合: **お手付き**。キューから **先頭を削除**し、**新しい先頭**に解答権が移る（説明は **同一ターン内** で継続）。

6. **解答順**（キューの並び）は **全参加者の画面に表示**する。



### 7.6 説明時間（MVP）



- **MVP** では、**各ターンの説明パート**に対する **アプリによる時間制限**および **時間切れ**による **強制ターン進行**は **行わない**。

- 説明の区切り・ターン進行は、**正解・不正解**（[7.5](#75-解答キューと正誤判定)）、**指摘フロー**（[7.4](#74-指摘と投票)）、および **参加者の運用**（通話上の合図等）に委ねる。

- **説明時間・時間切れ**を導入する場合は **将来機能**（[11.2](#112-説明時間時間切れ将来)）とする。



### 7.7 得点配分と正解時の加点



- **指摘・投票**による減点は [7.4](#74-指摘と投票) のとおり。

- **正解時の加点** — **両者加点**とする。

  - 正解した **解答者** に **+1 pt**。

  - **当該ターンの説明者** にも **+1 pt**。

  - 同一の正解判定につき、上記は **それぞれ1回ずつ**（重複加点しない）。



#### 7.7.1 正解時加点の比較検討（参考）



**本仕様の採用**は **説明者にも加点**（両者加点）であり、下表の **右列** に合致する。



| 観点 | 解答者のみ +1 | 説明者にも加点（採用） |

|------|-----------------|----------------|

| **説明のモチベーション** | 説明は「役目」になりがち。うまく説明しても得点に直結しない。 | 説明にも報いがあり、**全員が説明者のターンを真剣にやりやすい**。 |

| **協力と対立** | 解答争いは熱くなりやすいが、説明者は **他者の得点を伸ばす側** に回りやすい。 | 説明者も自分の順位を上げられるので、**説明と当ての両方**で盛り上がる。 |

| **順位の解釈** | 「当てた数」に近い順位になりやすい。 | 「当てた数」と「説明で当てさせた貢献」が混ざる。**説明の上手さ**が得点に入る。 |

| **カタカナ禁止の主戦場** | 説明は減点（指摘）のリスク中心。**加点は解答側**に寄る。 | 説明の成功もプラスになるため、**説明と指摘のバランス**が取りやすい。 |

| **同点・バランス** | 説明回数が全員同じなら、本質はクイズ速攻に寄る。 | 説明が上手い人が有利になりすぎる可能性（回数公平なら緩和）。 |

| **実装・理解のしやすさ** | 単純。 | 配点（同じ +1 か、説明は +0.5 等）を決める必要がある。 |



※ 左列（解答者のみ）は **採用しない**（検討時の対比用）。



### 7.8 ルーム参加・ホスト



#### 7.8.1 入室時のプレイヤー設定



- **ルーム入室**（ゲーム待機画面に参加すること）を完了する条件は、**プレイヤーネーム** の入力・検証の完了のみとする。**手描きアイコンの確定を待って入室をブロックしない**（アイコン確定まで入室できないと **参加者の把握が困難**になるため）。

- **プレイヤーネーム** — ゲーム中の **表示名**。**入室前**に入力する。**必須**。入力規約は [7.8.2](#782-プレイヤーネームの入力規約) に従う。

- **手描きアイコン** — **入室後**（ロビー・待機画面等）で、アプリ内の **描画UI**（キャンバス等）により **編集・確定・更新**する。未設定のあいだは **プレースホルダー**を表示してよい。**空白キャンバス（未描画）のまま確定してもよい**。詳細は [7.8.3](#783-手描きアイコンの扱い) を参照。

- **ホスト（ルーム作成者）** も、ルーム作成時に **プレイヤーネーム** を **必須**で入力する。**手描きアイコン** は **入室後**（作成直後の待機画面を含む）に、他参加者と同様に編集・確定する。



#### 7.8.2 プレイヤーネームの入力規約



一般的なオンラインサービスの **ニックネーム／表示名** に倣い、MVP では少なくとも次を **クライアントおよびサーバの両方**で検証する。



| 項目 | 仕様 |

|------|------|

| 必須 | **1 件必須**。未入力で入室完了は不可。 |

| 前後の空白 | 半角・全角の **先頭・末尾の空白は削除**（trim）してから検証・保存する。 |

| 空白のみ | trim 後の文字数が **0** の場合は **無効**（空白・改行のみの名前は不可）。 |

| 最大文字数 | trim 後 **32 文字以下**。文字数は **Unicode コードポイント**単位で数える（`𠮷` のようなサロゲートペアは 1 文字として数える等、実装言語の慣習に合わせる）。 |

| 最小文字数 | trim 後 **1 文字以上**。 |

| 制御文字 | **不可**（例: U+0000〜U+001F、改行を名前に含めない等。拒否または入力時に除去）。 |

| 同一ルーム内の重複 | **許可する**（識別は **プレイヤーネーム**・**手描きアイコン**（描画がある場合）・**内部の参加者 ID** で行う）。 |



※ 絵文字を含める場合も、上記 **32 コードポイント**の範囲に収める。



#### 7.8.3 手描きアイコンの扱い



- **描画UI** で **1枚の画像**としてアイコンを **編集・確定・更新**する。**入室後**（ロビー・待機画面）を主とし、**アイコン確定まで入室をブロックしない**。

- **空白キャンバス**（未描画、または消しゴム等で描画を全消しした状態）のまま **確定してもよい**。

- 未設定または空白の場合、参加者一覧・解答キュー・投票UI 等では **プレースホルダー**（デフォルトシルエット・アバター等）を表示してよい。識別の主たる手段は **プレイヤーネーム** および **内部の参加者 ID** とする。

- **ゲーム開始**に際し、**全員のアイコン確定済み**をシステム上 **必須としない**（プレースホルダーのまま開始してよい）。

- 細部（真っ白塗りと「無描画」の区別、画像フォーマット、**ゲーム中**の再編集の可否等）は実装時に定め、必要に応じ [10.3](#103-その他必要に応じて追記) に追記する。



**省力化**（プリセットスタンプのみ、テンプレート枠内のみの着色、画像ファイルの直接アップロードのみ等）は **本仕様のスコープ外**とし、[11. 将来機能](#11-将来機能本仕様のスコープ外の候補) で扱う。



| 項目 | 仕様 |

|------|------|

| 参加 | **ルームコード**入力と **URL** からの参加の **両方**を用意する。 |

| ゲーム開始 | **ルーム作成者（ホスト）**のみが開始操作できる。 |

| ホスト権限 | 原則 **ルーム作成者**がホスト。 |

| ホスト切断 | ホストが切断された場合、**ホスト権限を他参加者に自動移譲**する（移譲順序は実装で定める。例: 参加順・ランダム）。 |

| 手動移譲 | ホストは **任意の参加者**にホスト権限を **手動で移譲**できる。 |



（キック、強制終了、一時停止などの有無は必要に応じ [10. 未確定・要検討](#10-未確定要検討) で追記する。）



### 7.9 スコア・順位



- 上記および指摘・投票による加減点を蓄積する。

- **MVP** — 全ラウンド終了時点の **得点で順位**を付ける。**同点**の場合は **同順位**でよい（タイブレークなし）。

- **ライフ制を有効にした場合（将来）** — 進行中の **ポイント**・**ライフ**の扱い・順位の付け方は [11.1](#111-ライフ制将来) を参照する。

## 8. データ・コンテンツ



### 8.1 お題（ワードリスト）



- **MVP**: アプリに同梱またはサーバ側で保持する **固定リスト**。

- **将来**: ユーザー独自リストの登録・選択（詳細は別途）。



### 8.2 参加者プロフィール



- 各参加者に紐づく **プレイヤーネーム**（文字列、**入室時に必須**、[7.8.2](#782-プレイヤーネームの入力規約)）と、**手描きアイコン**（ラスタ画像。**入室後**に編集・確定。**内容は空白（未描画）でよい**。[7.8.3](#783-手描きアイコンの扱い)。例: PNG。クライアントでキャンバスから書き出し）。

- 当該セッション（ルーム）内で同期され、全員の画面に表示される（アイコン更新時は **リアルタイムに反映**する）。

- 形式・最大サイズ・解像度の上限は実装時に定め、必要に応じ [10.4](#104-技術アーキテクチャ) に記載する。

## 9. 非機能・技術方針（概要）



本節はポートフォリオ・学習目的のための大方針であり、具体技術は別ドキュメントまたはリポジトリで定義してよい。



| 項目 | 方針 |

|------|------|

| 同期 | **セッション型**: 1 ルーム = 1 ゲーム、**全員が同一状態を参照**するリアルタイム同期。 |

| クライアント | Web（PC・スマホ）。**各人 1 端末**を基本とした UI。 |

| 音声 | **本アプリは通話を提供しない**。Discord 等は利用者が別途利用する。 |

## 11. 将来機能（本仕様のスコープ外の候補）



- **2 人プレイモード**（投票なし、指摘の扱いを別ルール化など）。

- **説明時間・時間切れ**（各ターンの説明パートの制限、時間切れ時の強制ターン進行等）— [11.2](#112-説明時間時間切れ将来)。

- **ユーザー独自お題リスト**の登録・共有。

- **説明者向け**: 長時間無操作時の **アラート**・**自動不正解**。

- **音声を補助する**文字起こし・カタカナ検出（参考表示のみ、判定は人間優先など）— 採用時は別仕様。

- **アイコン描画の省力化**: プリセットスタンプ、テンプレート枠内のみの着色、画像ファイルの直接アップロードのみ等。MVP では [7.8.3](#783-手描きアイコンの扱い) に従い **空白キャンバス可**としているが、**描画UIを経由しない**登録方法は別仕様で定義する。

- **音声通話の本アプリ内完結**: 説明・雑談・投票時の会話などを、**Discord 等の外部ツールに頼らず**本アプリだけで完結させる（例: WebRTC によるルーム内ボイスチャット、必要に応じてミュート・話者表示等）。MVP では [2.2](#22-アプリの担当範囲)・[9](#9-非機能技術方針概要) のとおり **音声は外部**とし、インフラ・プライバシー・帯域は別途検討する。



### 11.1 ライフ制（将来）



ルーム作成時に **ライフ制あり**のように有効化する想定。**MVP では [§6](#6-ルーム作成設定)・[7.9](#79-スコア順位) のとおりライフは用いない。**



#### 二つのステータス



| ステータス | 役割 |

|------------|------|

| **ポイント** | **正解**等により加算する（減点は [7.4](#74-指摘と投票) 等の既存ルール）。 |

| **ライフ** | 初期値はルーム設定で与える（数値は実装時に定める）。**指摘成功**（**是認**、または投票で **可決**され、当該ターンの **説明者**がカタカナ違反とされた場合）に **説明者のライフを減少**させる（例: 1）。 |



#### 失格



- **ライフが 0** になった参加者は **その時点で失格**する。

- **得点（ポイント）がいくらであっても**、失格後は **通常の順位付けの対象外**とする（「失格」として扱う）。



#### 順位付け（ライフ制ありのとき）



- **失格していない**参加者のうち、まず **得点が高い**順に並べる。

- **得点が同じ**場合は、**残りライフが多い**方を **上位**とする（タイブレーク）。

- 失格者同士の順序や表示方法は実装時に定めてよい。



#### 既存ルールとの関係



- 指摘可決時の **得点減**（[7.4](#74-指摘と投票)）と **ライフ減**を **併用**するか、ライフ制では **どちらか一方**にするかは、実装・バランス調整時に決定する。



### 11.2 説明時間・時間切れ（将来）



ルーム作成時に **各ターンの説明パート**に対する時間制限（分／秒）を **設定可能**とする想定。**MVP では [7.6](#76-説明時間mvp) のとおり採用しない。**



- 時間制限の対象は **各ターンの説明パートのみ**とする（解答・投票の個別タイマーは別途検討）。

- **時間切れ**になった場合:

  - 当該 **ターン** の説明は **終了**し、**次のターン** に進む（**次の説明者** に交代し、**お題は新しいワードに更新**される）。

  - 同一ラウンド内の **最後のターン** の後は、**次のラウンド** の 1 ターン目へ進む（全ラウンド終了時はゲーム終了へ）。

- 細かい遷移（時間切れ直後に **解答キュー**をクリアするか等）は実装時に明記する。
